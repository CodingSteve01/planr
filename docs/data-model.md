# Data Model

All types. No TypeScript ‚Äî the code is plain JS ‚Äî so these are documentation conventions, not enforced interfaces.

## Top-level shape

```js
{
  meta: { name, planStart, planEnd, version, holidays? },
  teams: [Team],
  members: [Member],
  tree: [TreeItem],
  vacations: [Vacation],
  holidays: [Holiday]
}
```

Persisted as JSON (`planr_v2` key in localStorage, or mounted `.json` file) or as Markdown (mounted `.md` file). See [import-export.md](import-export.md) for format details.

## TreeItem

```js
{
  id,            // hierarchical dot notation ‚Äî "P1", "P1.2", "P1.2.3"
  name,          // display label, shown everywhere
  status,        // "open" | "wip" | "done"
  team,          // team ID (resolved to team.name in UI)
  best,          // optimistic effort in days (0 = unestimated)
  factor,        // complexity multiplier, default 1.5
  prio,          // priority, lower = higher. Default 2
  deps,          // [taskId] ‚Äî predecessors (can reference any node, resolved to leaves at scheduling time)
  assign,        // [memberId] ‚Äî assignees (usually 1; multiple means any of them can pick it up)
  progress,      // 0-100 on leaves, auto-cascaded on parents (never set manually on parents)
  note,          // short free-form text, shown in tooltips
  seq,           // legacy sequencing hint, optional
  parallel,      // legacy flag: bypass person capacity (kept for MD round-trip, not in UI)
  _depLabels,    // { depId: "label" } ‚Äî display labels for individual deps

  // Root-only fields (id has no dot):
  type,          // "goal" | "painpoint" | "deadline"
  severity,      // "critical" | "high" | "medium"
  date,          // target date (YYYY-MM-DD) for deadlines
  description,   // long-form description, richer than `note`

  // Scheduling overrides (on any item):
  decideBy,      // YYYY-MM-DD ‚Äî decision gate date (amber/red diamond in Gantt)
  pinnedStart    // YYYY-MM-DD ‚Äî hard floor for the scheduler
}
```

### ID conventions

- Roots: `P1`, `P2`, `D1`, `Pr1`, etc. ‚Äî no dot
- Children: `P1.1`, `P1.2`, `P1.2.3` ‚Äî dot-separated integers

Root IDs are user-chosen by convention (typically `P` for painpoint, `D` for deadline, etc.). Child IDs are auto-generated by `nextChildId(tree, parentId)` in [scheduler.js](../src/utils/scheduler.js).

### Leaf vs parent

A node is a leaf iff no other node has it as a prefix-parent. `isLeafNode(tree, id)` checks structurally ‚Äî there is no "level 3 = leaf" rule. Only leaves are scheduled.

### Progress cascade

- **Leaves**: explicit `progress` 0-100, or derived from status (`done` = 100, `wip` = 50, else 0)
- **Parents**: weighted average of leaf progress, weights = realistic effort (`best √ó factor`). Cascades up through all ancestors via `treeStats(tree)`.

## Member

```js
{
  id,     // stable member ID
  name,   // shown everywhere
  team,   // team ID
  role,   // free-form role text
  cap,    // capacity fraction: 1.0 = full-time, 0.5 = half-time
  vac,    // total vacation days per year (default 25)
  start   // availability start date, YYYY-MM-DD
}
```

**Capacity** ‚Äî `cap` scales the whole weekly budget. `vac` splits into explicit vacation weeks (zero-capacity) + a blanket deduction across the horizon. See [scheduler.md](scheduler.md#vacation-model).

**Short name** ‚Äî derived automatically for MD export (`buildMemberShortMap`). When the derived base collides, ALL occurrences get a numeric suffix so no one gets the bare base.

## Team

```js
{ id, name, color }
```

- `color` ‚Äî any CSS color (typically hex). Propagates into the tree, Gantt bars, and network graph node borders.
- Teams are managed under the **Resources** tab, not in Settings.

## Vacation

```js
{ person, week, note? }
```

- `person` = member ID
- `week` = Monday of the vacation week (YYYY-MM-DD)
- `note` = optional free-form text

Vacation weeks zero a person's capacity for that week. Use this for booked, planned time off. Use `member.vac` for the background "they have N more days they'll take sometime" deduction.

## Holiday

```js
{ date, name, auto }
```

- `auto: true` for entries generated by the NRW preset (Easter-based algorithm)
- `auto: false` for user-added entries

Holidays reduce the working-day count for affected weeks.

## meta

```js
{
  name,       // project name, shown in the title bar
  planStart,  // YYYY-MM-DD ‚Äî first week to schedule
  planEnd,    // YYYY-MM-DD ‚Äî last week to schedule
  version,    // "2" ‚Äî current file-format version
  holidays?   // "NRW" or similar preset marker (not exported to MD)
}
```

## Field defaults

When a field is missing from an import:

- `tree[].status` ‚Üí `"open"`
- `tree[].factor` ‚Üí `1.5`
- `tree[].prio` ‚Üí `2`
- `tree[].deps` ‚Üí `[]`
- `tree[].assign` ‚Üí `[]`
- `tree[].severity` ‚Üí `"high"` (only on roots with a type)
- `members[].cap` ‚Üí `1.0`
- `members[].vac` ‚Üí `25`
- `meta.planStart` ‚Üí today
- `meta.planEnd` ‚Üí year-end two years out
